# v0.3 Agent Memory Upgrade Design

## Purpose

Across Context v0.3 upgrades the product from a local memory vault with MCP tools
into a more complete agent memory governance layer. The release keeps the
current zero-runtime-dependency, local-first model while improving protocol
compatibility, retrieval quality, memory review, and public positioning.

## Product Goals

- Make Across Context feel current with the modern agent ecosystem without
  depending on hosted AI services.
- Improve memory retrieval quality enough that `semantic` and `hybrid` search
  produce explainable, stable recommendations.
- Let users review and govern automatic memories from both CLI/MCP and the
  local dashboard.
- Expose richer MCP surfaces so clients can discover project context and memory
  workflows without only calling tools.
- Make Agent Card output clearer for A2A-style discovery while keeping private
  paths out of public metadata.
- Document the intended agent workflow for Codex, Claude Code, Cursor, OpenAI
  Agents-style runtimes, and other MCP-aware tools.

## External Alignment

- MCP latest specification: tools remain the action surface, while resources and
  prompts provide discoverable read surfaces and reusable workflows.
  Reference: https://modelcontextprotocol.io/specification/latest
- A2A and MCP: A2A is useful for agent-to-agent task communication, while MCP is
  useful for exposing tools and contextual resources. Across Context should
  advertise that complement instead of pretending to replace either protocol.
  Reference: https://a2a-protocol.org/latest/topics/a2a-and-mcp/
- OpenAI Agents-style workflows emphasize tools, handoffs, guardrails, and
  tracing. Across Context should map memory search, memory write, and pending
  approval onto that workflow.
  Reference: https://platform.openai.com/docs/guides/agents
- LangGraph memory guidance separates memory into semantic, episodic, and
  procedural categories. Across Context already has memory types that map well
  to this model: notes/decisions/preferences, sessions, and commands.
  Reference: https://docs.langchain.com/oss/python/langgraph/memory
- Vercel AI SDK agents emphasize long-running, resumable, tool-using agent
  behavior. Across Context hooks and pending review should be documented as a
  lightweight local memory companion for these systems.
  Reference: https://ai-sdk.dev/docs/agents

## Scope

### In Scope

1. MCP read-surface upgrade
   - Add `resources/list` and `resources/read` support to the stdio server.
   - Add `prompts/list` and `prompts/get` support to the stdio server.
   - Add MCP definitions for resources and prompts in `src/mcp.js`.
   - Keep existing tools backward-compatible.

2. Structured and explainable search
   - Replace the current small related-term scorer with a local weighted
     retrieval module.
   - Keep `keyword`, `semantic`, and `hybrid` modes.
   - Add field weights for text, type, tags, project name, status, and recency.
   - Return match explanations with matched terms, matched fields, and score
     components.
   - Preserve deterministic behavior for tests.

3. Memory governance operations
   - Add vault support for batch status updates.
   - Add CLI support for JSON search explanations and batch memory status
     updates.
   - Add dashboard API endpoints for search and memory status changes.
   - Keep all writes going through the existing policy engine.

4. Dashboard workflow upgrade
   - Add search/filter controls in the rendered HTML.
   - Add per-row approve, archive, expire, and forget actions for local use.
   - Keep the dashboard dependency-free and server-rendered with small inline
     browser JavaScript.
   - Do not introduce a bundler or frontend framework.

5. Agent Card and documentation
   - Extend `agent-card` output with protocol version, discovery metadata,
     memory types, governance capabilities, and privacy posture.
   - Update README and examples to explain v0.3 workflows.
   - Avoid exposing absolute local paths in exported public metadata.

6. Tests and verification
   - Add focused tests for every behavior change.
   - Run `npm test` and `bash scripts/check.sh`.

### Out of Scope

- Cloud sync, hosted storage, accounts, or telemetry.
- External embedding APIs or heavyweight vector database dependencies.
- Replacing JSONL storage with SQLite or another database.
- Rewriting the dashboard as React/Vite/Next.
- Full A2A task server implementation.
- Secrets redaction beyond the existing policy engine plus incremental tests.

## Architecture

The current codebase already has clear boundaries:

- `src/vault.js`: storage and memory lifecycle.
- `src/memory-policy.js`: write admission and safety policy.
- `src/semantic-search.js`: retrieval scoring.
- `src/mcp.js`: MCP capability definitions.
- `src/mcp-server.js`: JSON-RPC stdio dispatch.
- `src/dashboard.js`: local HTTP dashboard and API.
- `src/agent-card.js`: public agent metadata.
- `src/cli.js`: user-facing commands.
- `src/exporters.js`: generated agent instructions.

v0.3 keeps these boundaries. Search logic remains isolated in
`src/semantic-search.js`; the vault calls it but does not know scoring internals.
MCP capability definitions stay in `src/mcp.js`; the stdio server only dispatches
methods. Dashboard operations call the vault instead of modifying JSONL files
directly.

## Data Flow

### Search

1. CLI, MCP tool, hook, or dashboard sends a query.
2. `ContextVault.search()` loads global and project memories with optional
   status and visibility filters.
3. `searchEntries()` scores each entry using token overlap, related terms,
   field weights, status/type/recency boosts, and mode-specific behavior.
4. Results include `{ entry, score, matchMode, explanation }`.
5. Callers render the explanation when useful and ignore it when backward
   compatibility matters.

### Governance

1. Automatic note/session memories still default to `pending`.
2. CLI/MCP/dashboard can list pending memories.
3. Single or batch status changes call vault lifecycle methods.
4. Status changes update `updatedAt` and rewrite only the affected JSONL file.
5. Team export continues to include only team-visible project memories.

### MCP Discovery

1. `initialize` advertises tools, resources, and prompts capabilities.
2. `resources/list` exposes stable URIs such as:
   - `across-context://agent-card`
   - `across-context://stats`
   - `across-context://memories`
   - `across-context://project-context`
3. `resources/read` returns text or JSON content for those URIs.
4. `prompts/list` exposes reusable prompts such as:
   - `task-start-context`
   - `task-end-summary`
   - `memory-review`
5. `prompts/get` returns prompt messages with caller-provided arguments.

## User Experience

### CLI

Existing commands continue to work. Search output can stay concise by default,
with JSON or explanation output added only where the codebase pattern supports
it cleanly.

### MCP Clients

Existing tool names remain unchanged. New resources and prompts make Across
Context easier for clients to inspect without making arbitrary tool calls.

### Dashboard

The dashboard remains local, plain HTML, and dependency-free. It gains:

- A search box.
- Status/type/scope filters.
- Result counts.
- Inline lifecycle buttons.
- Short match explanation text when searching.

## Error Handling

- Unknown MCP methods return the existing JSON-RPC error shape.
- Unknown resource or prompt names throw clear errors.
- Invalid memory status values use the existing status validation.
- Dashboard mutation endpoints accept only `POST`.
- Dashboard JSON responses include a simple error payload and status code for
  invalid requests.
- Search with an empty query returns recent filtered memories instead of
  failing in dashboard contexts; CLI search keeps requiring a query.

## Privacy and Safety

- The default vault remains under `~/.across/data/across-context`.
- No hosted service calls are added.
- No private absolute paths are included in Agent Card public output.
- Memory writes still pass through `MemoryPolicyEngine`.
- Dashboard binds to `127.0.0.1` by default.
- Team export remains visibility-filtered.

## Testing Strategy

- Unit-test search scoring, explanations, filters, and ranking.
- Unit-test vault batch status updates.
- Unit-test MCP resources and prompts definitions.
- Stdio-test MCP `resources/list`, `resources/read`, `prompts/list`, and
  `prompts/get`.
- Unit-test dashboard search and mutation APIs.
- CLI-test any new or changed command behavior.
- Run the full test suite and package check script before completion.

## Acceptance Criteria

- All existing tests continue to pass.
- New MCP clients can discover tools, resources, and prompts.
- `hybrid` search ranks exact task matches above related-term-only matches and
  returns an explanation.
- Pending memory review can be completed from the dashboard API.
- Agent Card output describes governance and protocol readiness without leaking
  local paths.
- README explains v0.3 workflows in English and Chinese.
- `npm test` and `bash scripts/check.sh` pass.
